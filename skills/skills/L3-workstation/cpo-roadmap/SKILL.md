# 📍 产品路线图

## 触发条件
- 🕐 定时：每周一 09:00 检查进度
- 季度规划时
- 紧急需求插入时评估影响

## 输入要求
1. 需求池（含优先级）
2. CEO 战略方向
3. CTO 技术评估（工期、依赖）
4. 可选：客户承诺、竞品节奏

## 执行步骤
1. **需求排序**：按战略价值×用户影响×技术成本排序
2. **版本规划**：分配到各版本/Sprint
3. **依赖梳理**：标注前后依赖关系
4. **进度追踪**：每周检查偏差
5. **变更管理**：新需求插入时评估代价

## 输出模板
```
## 产品路线图 Q{N}

### 本季度目标
{1-2 句话}

### 版本规划
| 版本 | 时间 | 核心功能 | 状态 |
|------|------|---------|------|
| v{X} | {月} | | 🟢开发中/☐待开始 |

### 本周进度
| 功能 | 计划 | 实际 | 偏差 | 风险 |
|------|------|------|------|------|
| | | | | |

### 变更记录
| 日期 | 变更 | 原因 | 代价 |
|------|------|------|------|
| | | | |

### 下周关注
- {重点事项}
```

## 异常处理
- 紧急需求插入 → 先判断 bug（转 CTO）还是产品需求；插入时明确代价"加这个就得砍那个"
- 进度落后 → 预警 CEO，给调整建议（砍范围/加资源/延期）
- CEO 需求朝令夕改 → 用变更记录帮 CEO 看到"改了多少次"

## 边界
- 不做技术工期评估（→ CTO）
- 不做资源调配（→ COO）
- 不做战略方向决策（→ CEO / CSO）
