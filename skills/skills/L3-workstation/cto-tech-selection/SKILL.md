# 📍 技术选型对比

## 触发条件
- CEO 问"用 A 还是 B？"
- 新项目启动需要技术选型
- 现有技术方案需要替换

## 输入要求
1. 候选技术方案（至少 2 个）
2. 业务需求（要解决什么问题）
3. 团队能力档案（谁来做）
4. 可选：预算约束、时间要求

## 执行步骤
1. 明确评估目标：选型是为了解决什么问题？
2. 按六维框架逐一评估每个方案：
   - **功能满足度** — 能不能做到需求的 100%？80%？
   - **工程成本** — 多少人、多少天能上线？
   - **运维成本** — 上线后每月花多少？谁维护？
   - **扩展性** — 用户/数据翻 10 倍还能撑住吗？
   - **团队匹配** — 现有团队的经验和学习成本
   - **风险** — 最坏情况是什么？能回退吗？
3. 搜索最新信息验证（社区活跃度、已知问题）
4. 给出明确推荐 + 理由
5. 如果差距不大 → 明确说"差别不大，选团队更熟的"

## 输出模板

```markdown
# 技术选型对比 | [主题]

## 需求确认
[要解决什么问题，关键约束是什么]

## 方案对比

| 维度 | 方案 A | 方案 B | 方案 C |
|------|--------|--------|--------|
| 功能满足度 | ⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| 工程成本 | 2人×3月 | 1人×2月 | 3人×4月 |
| 运维成本 | ¥5k/月 | ¥2k/月 | ¥8k/月 |
| 扩展性 | 中 | 高 | 高 |
| 团队匹配 | ✅ 熟悉 | ⚠️ 需学习 | ❌ 无经验 |
| 风险 | 低 | 中 | 高 |

## 推荐：方案 [X]

**理由：**
1. [核心理由]
2. [次要理由]
3. [风险可控]

**风险：**
- [主要风险] → [缓解措施]

## 不推荐的原因
- 方案 Y：[为什么不选]
```

## 异常处理
- 超出知识范围 → "这个领域我不够熟，建议找专业顾问"
- CEO 给的信息不够 → 列出需要确认的问题

## 边界
- 不做最终决策（CEO 决定）
- 不做预算审批（→ CFO）
- 不做产品需求确认（→ CPO）
