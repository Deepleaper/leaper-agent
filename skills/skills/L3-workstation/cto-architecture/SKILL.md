# 📍 架构评审

## 触发条件
- CEO 提新需求/战略变化影响架构
- 新技术冲击现有架构
- 产品规模增长触发架构瓶颈

## 输入要求
1. CEO 描述的需求或变化
2. 现有技术栈档案
3. 可选：预期用户规模、性能要求

## 执行步骤
1. 理解需求：确认要解决什么业务问题
2. 评估现有架构是否支撑新需求
3. 如需新架构，设计方案（至少 2 个对比）
4. 评估每个方案：
   - 优势
   - 风险点（单点故障、性能瓶颈、数据一致性）
   - 改造成本（人力×时间）
   - 对现有系统的影响
5. 给出推荐方案和理由
6. 列出下一步行动清单

## 输出模板

```markdown
# 架构评审报告 | [主题]

## 背景
[业务需求/技术变化描述]

## 现状分析
[现有架构能/不能支撑的原因]

## 方案对比

| 维度 | 方案 A | 方案 B |
|------|--------|--------|
| 架构描述 | ... | ... |
| 优势 | ... | ... |
| 风险 | ... | ... |
| 成本（人×月） | ... | ... |
| 扩展性 | ... | ... |

## 推荐
**方案 [X]**，理由：[1] [2] [3]

## 风险缓解
- 风险1 → 缓解措施
- 风险2 → 缓解措施

## 下一步
1. [具体行动]
2. [具体行动]
```

## 异常处理
- 信息不足 → 列出需要 CEO 补充的信息清单
- 超出能力 → "这个架构复杂度需要找资深架构师 review"

## 边界
- 不做产品需求定义（→ CPO）
- 不做成本预算决策（→ CFO）
